# Anvil pub-grammar migration notes (canary)

Canary evidence from migrating the `tractatus-world-models` follow-up paper thread
(v1 â†’ v2) into the freshly-installed anvil `pub` skill grammar. This is a pure FORMAT
migration; every point below is friction encountered adapting an existing
hand-authored paper to anvil's conventions. Intended as upstream feedback for the
anvil `pub` skill.

## F1. Default class `anvil-paper.cls` is citation-style-incompatible with numeric-cite papers

`templates/anvil-paper.cls` loads natbib with `[round,sort&compress,authoryear]`. Any
existing paper written with plain numeric `\cite{}` (as ours is) cannot adopt the class
without silently switching every in-text citation from `[7]` to `(Author Year)` â€” a
visible rendered-content change. For a migration whose contract is "content-identical,"
the class is unusable. We kept `\documentclass[10pt]{article}` and used
`\bibliographystyle{plain}`.

- **Impact:** the SKILL.md draft procedure says "uses `anvil-paper.cls` by default." A
  migration path (as opposed to a fresh draft) has no supported way to keep numeric
  citations while adopting the class.
- **Suggested upstream fix:** either (a) a class option `\documentclass[numeric]{anvil-paper}`
  that swaps `authoryear`â†’numbers, or (b) explicit guidance in a "migrating an existing
  paper" section that keeping `article` + a note is the sanctioned deviation.

## F2. `anvil-paper.cls` bakes a preamble that collides with custom paper preambles

Our paper defines a `lstdefinelanguage{lean}` listing style, custom theorem
environments (`theorem`/`lemma`/`definition`/`corollary`/`remark` with `[section]`
numbering), and `\newcommand`s (`\lean`, `\TLP`). The class already `\RequirePackage`s
amsthm, xcolor, hyperref, caption with its own `\captionsetup`, and redefines
`\section`/`\abstract`. Merging our preamble onto the class risks option clashes
(hyperlink colors, caption fonts) and duplicate-definition warnings. Keeping `article`
sidesteps all of it. A real paper's preamble is load-bearing; the minimal class assumes
a paper with no custom preamble.

## F3. `refs.bib` conversion from inline `thebibliography` is manual and lossy-prone

The 17 `\bibitem`s had to be hand-converted to `@book`/`@article`/`@inproceedings`/
`@unpublished` entries. Judgment calls with no mechanical answer:

- **Entry type inference.** "Translated by â€¦ London: Routledge" â†’ `@book` with a
  `translator` field; a `PAS` supplementary volume â†’ `@article`; ECAI/CADE proceedings â†’
  `@inproceedings`; "Manuscript under review" (the companion) â†’ `@unpublished`; arXiv
  (`evans2014`) â†’ `@article` with the arXiv id folded into `journal` (BibTeX `plain`
  has no native arXiv field).
- **No field to hold `note`-like prose.** v1's "First published 1921 as â€¦" and
  "Supplementary Volume 9" had to be squeezed into `note`/`volume`, which `plain.bst`
  renders differently from the hand-typed bibitem. Rendered output is *equivalent*, not
  byte-identical.
- **DOIs preserved** where v1 had them (moss2012, button2017, spinney2022, weiss2017);
  `plain.bst` does not print `doi` fields, so they are carried but invisible â€” matches
  v1, which also did not render DOIs as links (they were plain text in the bibitem).
- **No litsearch sibling / `candidates.bib`** existed to convert from; the bib was born
  inline. Anvil assumes bib entries flow from `refs.bib` or a litsearch `candidates.bib`;
  a paper that predates the grammar has neither.

## F4. `_progress.json` is gitignored in this consumer repo

Repo `.gitignore` line ~27 (`_progress.json`) ignores the file the anvil grammar treats
as a per-version resume hint. We wrote it for runtime use and force-added it to the
commit (`git add -f`) so the version dir is self-describing on checkout. Anvil's own
contract calls `_progress.json` "a resume hint, not a source of truth" (validation is by
file existence), so the ignore is arguably fine â€” but a consumer adopting anvil should
decide explicitly whether to un-ignore it. Flagged for the repo owner.

## F5. `_progress.json` schema: iteration numbering for a migration

The schema's `metadata.iteration` is documented for the fresh draftâ†’reviewâ†’revise loop.
For a format migration that produces `.2/` without a `.1.review/`, "iteration" is
ambiguous. We set `iteration: 2` (matches the version-dir number) and added a
non-schema `metadata.migration` extension block (allowed: "Skill-specific extensions
are allowed â€¦ the merge rule preserves any extension fields"). No `phases.draft` sub-key
other than `draft` was set, since figures were copied, not generated by `pub-figures`.

## F6. No Synthese venue YAML ships; `venue` key omitted from `.anvil.json`

`.anvil/skills/pub/rubrics/` ships only `arxiv.yaml`, `nature.yaml`, `neurips.yaml`.
There is no `synthese.yaml`. Per SKILL.md, setting `venue: synthese` with no matching
YAML in any of the three discovery tiers makes `pub-review` emit a stdout warning and
fall back to the generic /44 â€” advisory-only, non-blocking. To avoid a spurious warning
on the canary, we **omitted** the `venue` key from `.anvil.json` (kept only
`max_iterations: 4`) and recorded the venue (Synthese) in `BRIEF.md` frontmatter
instead, where it is documentation rather than a rubric selector. A consumer wanting a
scored Synthese overlay would need to drop a `Rubric`-shaped `synthese.yaml` into
`.anvil/skills/pub/rubrics/` (or the per-thread `<thread>/.anvil/rubrics/`).

## F7. Filename grammar: `paper.tex` â†’ `main.tex`, `data/` â†’ `figures/src/`

Two mechanical renames the grammar mandates:
- Entry point must be `main.tex` (v1 used the repo-wide convention `paper.tex`, matching
  the sibling `tractatus-ontology.N/paper.tex` threads).
- Figure source scripts must be under `figures/src/` (v1 used a sibling `data/` dir).

Both are harmless here but mean this thread's on-disk layout now diverges from the
older `tractatus-ontology` paper threads in the same repo â€” the repo now has two
paper-layout conventions side by side until the older threads migrate too.

## F8. Compiled `paper.pdf` is NOT committed at DRAFTED state

SKILL.md assigns the PDF-compile-and-commit responsibility to `pub-audit` (which runs
at READY), not to the drafter. So the DRAFTED version dir contains `main.tex` +
`refs.bib` + `figures/` + `_progress.json` + `changelog.md`, but **no `paper.pdf`**.
This is a deliberate contrast with the repo's existing `tractatus-ontology.N/` threads,
each of which commits `paper.pdf`. We verified compilation succeeds cleanly (9 pages,
0 undefined refs/cites) but left no PDF in the version dir, per the contract. Consumers
used to committing the PDF at draft time will find this surprising.

## Summary

The grammar fits a *fresh* anvil-native paper cleanly. The friction is concentrated
where a *pre-existing* hand-authored paper meets the defaults: the default class's
author-year natbib (F1) and baked preamble (F2) are the sharpest â€” a migrating numeric-
cite paper with a custom preamble simply cannot use `anvil-paper.cls` without content
changes, so "keep `article`, note the deviation" should be a first-class documented
migration path, not an escape hatch. F4 (gitignored progress) and F8 (no draft-time PDF)
are consumer-repo-convention clashes worth an explicit adoption decision.

## Audit-phase friction (pub-audit run, 2026-07-13)

## F9. `.anvil` venv was not pre-synced; first `uv run` builds it on the fly

The first `uv run --project .anvil python -m anvil.lib.sidecar ...` invocation created
the venv from scratch ("Creating virtual environment at: .anvil/.venv ... Built anvil")
and emitted a hardlink warning because the uv cache lives on a different filesystem
than the `/Volumes/Stripe` worktree ("Failed to hardlink files; falling back to full
copy"). Harmless, but a critic dispatched parallel-safe could race on first-sync;
consider `uv sync --project .anvil` in worktree setup, and `UV_LINK_MODE=copy` for
worktrees on external volumes.

## F10. `python -m anvil.lib.sidecar` emits a runpy RuntimeWarning on every call

Every CLI-shim invocation prints `<frozen runpy>:130: RuntimeWarning:
'anvil.lib.sidecar' found in sys.modules after import of package 'anvil.lib' ...`.
Cosmetic (exit codes and behavior are correct: stage/cleanup/commit all worked, and
the commit rename was atomic), but it is noise in every audit log. Upstream fix:
a console-script entry point instead of `-m`, or move the CLI into a `__main__`-only
module.

## F11. No `<thread>/refs/` dir; claim-support ground truth lives in the v1 dir

The pub-audit contract reads author-supplied sources from `<thread>/refs/**`. This
migrated thread has none â€” the verified citation data (DOIs, volumes, pages, claim
annotations) lives in pre-anvil `tractatus-world-models.1/literature.md`. The audit
used it as ground truth by instruction, but a by-the-book auditor would have marked
every citation `unverified`. Migration guidance should say: carry `literature.md`
(or the source notes it summarizes) into `<thread>/refs/` so the audit phase can see it.

## F12. External-artifact claims (Lean) are outside the audit contract's checklists

The paper's central verifiability claims (63 declarations, no sorries, axiom
footprint, warning-free build) concern the repo's Lean development, not cites or
figures. pub-audit's procedure has no step for external proof artifacts; this repo's
verify-artifact-claims norm filled the gap (grep of all 42 Appendix cross-references,
full `lake build` in the worktree â€” which required a fresh ~3,000-job mathlib cache
fetch since `.lake/` is per-worktree â€” and `#print axioms` on 10 headline theorems).
That check is what surfaced the run's one critical flag (Theorem 4.3 mis-attributed to
`horn_realizable_iff`), which no cite/figure/build check would have caught. Suggest an
optional "artifact claims" step in pub-audit for papers whose evidence inventory is a
formal development.

## Review-phase friction (pub-review run, 2026-07-13)

## F13. Render-gate input naming: command doc says `paper.pdf`, audit artifact is `main.pdf`

`commands/pub-review.md` step 4b names the gate input `pdf_path: <thread>.{N}/paper.pdf`
"produced by pub-audit", but the audit's build artifact for a `main.tex` entry point is
`main.pdf` (this thread's audit committed `tractatus-world-models.2/main.pdf`). Followed
literally, the gate would either fail open ("paper.pdf absent â€” run pub-audit first",
despite the audit having run) or spuriously fail the `compile` dimension ("PDF not
produced"). The reviewer gated `main.pdf` and recorded the mismatch here. Upstream fix:
step 4b should reference the entry-point-derived name (`main.pdf`) or the audit contract
should pin the output PDF name; the two docs currently disagree.

## F14. Multi-pass `compile-log.txt` triple-counts overfull boxes in the render gate

The audit's `compile-log.txt` concatenates the full `pdflatex â†’ bibtex â†’ pdflatex â†’
pdflatex` cycle, so `render_gate._parse_overfull_boxes` counted each unique overfull
box once per LaTeX pass: 18 hits for 6 unique boxes (worst 121.5pt). Verdict-wise it
made no difference (any hit over threshold fails the dimension), but the reported count
is inflated 3x and a reviser chasing "18 boxes" will find 6. Upstream fix: gate should
dedupe by (source line, amount), or the audit contract should specify capturing only
the final pass's log.

## Revise-phase friction (pub-revise run, 2026-07-13)

## F15. The fixed compile pass count (`pdflatex â†’ bibtex â†’ pdflatex Ă—2`) does not guarantee convergence

For v3 the standard cycle ends with LaTeX's "Labels may have changed. Rerun to get
cross-references right" on the final pass: the second post-`bibtex` pass is the first
one typeset with resolved numeric citations (`[?]` â†’ `[16]`), which reflows enough
lines to move several page breaks, so label *pages* recorded in the `.aux` shift once
more and a third post-`bibtex` pass is needed to stabilize (verified: pass-4 `.aux` is
byte-identical to pass-3's, and pass 4 is warning-free). Any numeric-cite paper whose
citation resolution changes line breaks near a page boundary will hit this. Impact:
an audit that runs the contract's literal pass count and greps the log for warnings
will see a benign rerun hint on an otherwise clean document; a `\pageref`-using paper
could even ship a stale page number. Upstream fix: make the compile contract "run
until the rerun warning disappears (max 5 passes)" â€” latexmk semantics â€” instead of a
fixed count.
